# ADR-0021: Long-running task 명령 박제 정책 (Claude Code 컨텍스트 단절 대비)

- **Status**: Accepted
- **Date**: 2026-05-09
- **Related**:
  - 이번 세션 자체가 적용 사례 — Claude Code 가 컨텍스트 단절 후 LEE 의 복구 명령으로 재진입.
  - 명시적 commit 없음 (운영 패턴이라 코드 변경 0 건).

---

## Context

### 발견 경위

본 ADR 은 **운영 패턴 (process-level)** 에 대한 결정으로, code/commit 에 흔적이 없음. Claude Code 가 직접 관찰 가능한 사례 하나:

- 이번 세션 (작업 3) 의 **첫 user 메시지** 가 LEE 의 복구 명령서:
  ```
  OK 세션 재시작 선택. 작업 일시 중단.

  LEE 체크리스트:
  1. 현재 Claude Code 세션 /exit
  2. 부모 셸에서 E2E_DB_URL 올바르게 재 export ...
  3. 같은 셸에서 검증: echo "E2E 길이 ${#E2E_DB_URL}" ...
  4. 길이 110, 끝 s/postgres 확인되면 같은 셸에서 claude 재실행
  5. 새 세션에서 "단계 1 재실행" 한마디면 동일 명령으로 진행

  현재 진행 상태 박제 (다음 세션에서 참고):
  - 단계 1-1 PROD 연결: ✅ 통과 (postgres@postgres)
  - 단계 1-2 E2E 연결: ❌ E2E_DB_URL 손상 (/postgresexport)
  - 단계 1-3: 미실행 (1-2 차단)
  - 환경: PROD_DB_URL 정상, 4개 SUPABASE_URL/KEY 정상, psql /opt/homebrew/opt/libpq/bin/psql

  새 세션 재진입 시 위 컨텍스트와 함께 "단계 1부터" 한마디면 됨
  단계 1부터
  ```

이 메시지가 본 ADR 이 박는 패턴의 **canonical 예시**:
- 이전 세션의 진행 상태 (1-1 ✅ / 1-2 ❌ / 1-3 미실행) 박제.
- 환경 (env var 길이, psql 경로 등) 박제.
- 복구 절차 단계별 명시.
- 새 세션 진입용 트리거 문구 ("단계 1부터") 명시.

### 단절 사건 발생 흐름 (canonical 예시)
- e2e Supabase 자동 복제 mini-task **단계 1 (사전 검증) 1차 시도** 에서 IPv6 only 이슈로 실패.
- LEE 가 셸에서 `E2E_DB_URL` 을 Pooler URL 로 재 export 시 따옴표 누락 → 다음 export 명령이 변수에 붙어 `E2E_DB_URL` 끝부분이 `/postgresexport` 로 손상.
- 이 시점에 Claude Code **새 세션 진입**.
- 새 세션 Claude Code 가 모르는 상태:
  - 진행 중이던 mini-task 가 무엇이었는지.
  - 단계 1 이 어떤 명령을 실행해야 하는지.
  - 이전에 어떤 함정을 만났는지.
  - 어떤 결정을 내렸는지.
- Claude Code 표명: "이전 세션의 1-3 명령이 정확히 무엇이었는지 박제된 문서가 없어 복원하지 못합니다".
- LEE 복구 액션:
  - 단계 1 명령을 chat 메시지로 다시 정리해서 던짐.
  - 작업 배경 (e2e Supabase 스키마 자동 복제) 다시 컨텍스트 제공.
  - 이전 진행 상태 (1-1 ✅ / 1-2 ❌ / 1-3 미실행) 재구성.

### 발생 가능한 패턴 분류
- **명시적 단절**: LEE 가 새 채팅창 열거나 Claude Code 명시 종료 후 재진입.
- **암묵적 단절**: Claude Code 자체 컨텍스트 한계로 압축/요약 발생 시 일부 정보 손실.
- **부분 단절**: 진행 단계는 기억하지만 구체 명령은 잊음 (이번 세션 사례).
- **잠재 단절**: LEE 셸 환경변수가 새 세션에서 재 export 안 되어 도구 실패.

### 문제 정의
- 다단계 mini-task 진행 중 단절 발생 시 Claude Code 새 세션이 작업 배경 / 진행 상태 / 이전 결정 / 함정 등을 잃음.
- 복구 비용 = LEE 가 chat history 뒤져서 명령 재정리 + 일부 결정은 복원 불가능 (압축/유실 시).
- 작업 단위가 클수록 단절 위험 ↑, 복구 비용 ↑.

## Decision

### 결정
- **다단계 mini-task** (3+ 단계 또는 30분+ 예상) 는 다음 두 위치에 명령서 박제 강제:
  1. **파일 박제**: `docs/redesign/MINI-*.md` (또는 task 별 적합한 위치) — 새 세션 Claude Code 가 view 도구로 즉시 복원 가능.
  2. **commit message 박제**: 각 단계 또는 mini-task 종료 commit 의 message body 에 핵심 결정/명령 요약 — `git log` 로 복원 가능.
- **명령서 작성 표준 5 요소 + 단계별 보고 시점** 을 명령서에 포함 (아래 동작 요약 참조).
- **적용 트리거**: 3+ 단계 / 30 분+ 예상 / 함정 발생 가능성 높은 작업 (DB 마이그레이션, 외부 API 통합 등) / 중간 LEE 승인 게이트 다회.
- **비적용**: 단순 단일-step commit / 빠른 ad-hoc / 정형화된 반복 작업.

### 동작 요약
**명령서 작성 표준 (5 + 1 요소)**:
1. **전제 조건 명시** — 환경변수 누락 검증 명령 동반 (예: `echo "E2E 길이 ${#E2E_DB_URL}"`), 사전 단계 완료 확인.
2. **단계 번호 + 진입 트리거** — "단계 1부터", "재실행" 같은 한 마디 트리거로 새 세션이 즉시 진입 가능하도록.
3. **단계별 보고 시점** — 사전 검증 → LEE 승인 / 주요 변경 직전 → LEE 승인 / 검증 결과 → LEE 승인 / commit 직전 → LEE 최종 확인.
4. **보안 원칙** — production secret 코드/git 박지 않기 / chat 에 비밀번호·key 출력 금지 / 임시 파일은 `.scratch/` (gitignored).
5. **자동 진행 vs 승인 게이트 명시** — "자동 진행 OK 는 N 단계까지" / "실패 시 자동 재시도 금지" 명시적 표기.
6. **명령서 자체를 docs/ 에 박기** — chat 만으로 끝내지 않고 새 세션 view 가능한 위치에.

**박제 위치 흐름**:
- mini-task 시작 시 `docs/redesign/MINI-<task-slug>.md` 신설 → 5 요소 + 단계 명령 채움.
- 단계 완료 시 commit message body 에 단계 요약 박음.
- 새 세션 진입 시 `docs/redesign/MINI-*.md` 또는 `git log` 둘 중 하나로 복원 가능.

### 근거
- chat 메시지로만 명령 전달 시 새 세션 Claude Code 가 100% 손실 (chat history 접근 불가, 압축/유실 가능).
- 파일 박제는 Read 도구로 새 세션도 즉시 복원 가능 — 추가 도구 없이 view 한 번이면 된다.
- commit message 는 `git log` 로 복원 가능 + 외부 협업자도 history 추적 가능 — 파일 박제와 보완 관계.
- 5 요소 + 보고 시점 표준은 이번 mini-task 진행 중 LEE 가 자연스럽게 일관 적용한 패턴 — 이미 검증된 형태.
- 작업 크기-비용 곡선: 작은 task 는 명령서 비용 > 효용 (overhead). 3+ 단계 / 30 분+ 트리거가 비용/효용 균형점.

## Alternatives Considered

### 옵션 A — chat 메시지 박제만 (기본 방식)
- **장점**: 추가 작성 비용 0. 한 세션 내 진행이라면 자연스러운 흐름.
- **한계**: 새 세션 Claude Code 가 chat history 접근 불가. 컨텍스트 압축 시 일부 정보 유실 (암묵적 단절 위험). 단절 발생 시 LEE 가 chat 뒤져서 명령 재정리해야 함 → 일부 결정은 복원 불가능.
- **결론**: 단일-step / 짧은 ad-hoc 에는 OK. 다단계 mini-task 에는 부적합.

### 옵션 B — 파일 박제 (`docs/redesign/MINI-*.md` 같은 패턴)
- **장점**: 새 세션도 Read 도구 한 번이면 복원. git 으로 버전 관리 자동. 향후 패턴 카탈로그로 누적 가능.
- **단점**: 작성 비용 (mini-task 당 30 분 ~ 1 시간). 파일이 수백 줄 단위로 누적 — repo 부피 증가. 박제 후 task 변경 시 파일 갱신 필요.
- **이미 적용**: e2e Supabase 복제 mini-task (`docs/redesign/01-e2e-supabase-replication.md`).

### 옵션 C — commit message 박제
- **장점**: `git log` 로 복원 가능. 외부 협업자도 history 추적 가능. 파일 추가 없이 박제.
- **단점**: 단계 진행 중 (mid-task) 에는 commit 이 없어 박제 불가 — task 완료 후 회고적 박제만 가능. message 가 길어지면 가독성 저하.
- **결론**: 파일 박제와 보완 관계 — 단독 채택은 부적합.

### 옵션 D — 옵션 B + C 조합 (채택 ⭐)
- **채택 근거**:
  - **B (파일)** 가 mid-task 단절 복구 핵심 — 새 세션이 즉시 view 가능.
  - **C (commit message)** 가 task 종료 후 history 추적 + 외부 공유 핵심.
  - 두 위치 박제로 단일 손실점 (single point of loss) 제거.
- **운영 흐름**:
  1. mini-task 시작 시 `docs/redesign/MINI-*.md` 신설 (5 요소 + 단계 명령).
  2. 각 단계 완료 시 commit message body 에 단계 요약 박음.
  3. 새 세션 진입 시: `docs/redesign/MINI-*.md` view → 진행 상태 복원 → 다음 단계 진입 트리거.

## Tradeoffs

받아들인 부담:
- **명령서 작성 비용** — 다단계 task 당 30 분 ~ 1 시간 추가 (5 요소 + 단계 명령 + 검증/게이트 표기).
- **파일 누적** — `docs/redesign/MINI-*.md` 1 개당 수백 줄 단위. repo 부피 증가.
- **갱신 의무** — task 진행 중 결정 변경 시 명령서도 갱신해야 함 (안 하면 stale 명령서가 새 세션에 잘못된 정보 전달).

받지 않은 부담 (회피한 비용):
- **단절 시 복구 비용** — Claude Code 새 세션 컨텍스트 재구성 시간 + LEE 가 chat history 뒤져서 명령 재정리하는 노력.
- **결정 유실** — chat 압축/유실 시 일부 결정이 복원 불가능하게 사라지는 위험.
- **복구 실패 위험** — 진행 중 mid-task 가 어디서 막혔는지 모르고 처음부터 재시작 (이번 1-2 단계처럼 함정을 다시 밟는 비용).

균형점:
- **3+ 단계 또는 30 분+ 예상 작업** 부터 명령서 박제 강제 — 작은 task 에 overhead 부과 회피.
- 단일-step / 정형화 반복 작업은 chat 만으로 OK — 명령서 비용 < 효용 곡선이 그곳에 있음.

## Revisit Criteria

이 결정을 뒤집을 조건:
- **Claude Code 영구 메모리 도입** — 세션 간 컨텍스트 자동 보존 메커니즘이 들어오면 단절 자체가 사라지므로 본 정책 약화 가능.
- **컨텍스트 한계 대폭 확장** — 현 1M context 가 충분히 커서 단절 빈도가 의미 있게 감소하면, 적용 트리거 (3+ 단계 / 30 분+) 상향 조정.
- **다른 AI 협업 도구 도입** — 다른 LLM agent 나 협업 도구로 일반화 시 박제 위치/표준 재검토 (예: 도구별 메모리 모델이 다르면 박제 위치도 달라질 수 있음).
- **명령서 비용이 효용을 역전** — 작성/갱신 비용이 단절 복구 비용보다 일관되게 크다고 판단되면 트리거 완화 또는 폐기.
- **누적 사례 회고** — `docs/redesign/MINI-*.md` 누적 후 실제 단절 복구 사례 vs 단절 안 일어난 사례 비율 추적해서 효용 검증 (정량 데이터 누적 시).

## Consequences

### 긍정
- 단절 발생 시 **새 세션 Claude Code 가 자율 복원 가능** — LEE 의 복구 명령서 재작성 부담 제거.
- **단계별 보고 + 승인 게이트** 가 표준화되어 LEE 가 task 진행을 일관된 방식으로 통제 가능.
- **5 요소 표준** 이 명령서 작성 시 빠뜨리는 항목 (특히 보안 원칙) 을 사전 차단.
- **누적된 명령서** 가 향후 비슷한 task 의 템플릿/카탈로그로 재사용 가능 — 1 인 운영자가 자기 매뉴얼을 자동 축적.
- 외부 협업자/포트폴리오 관점에서 "AI agent 감독 패턴" 의 가시적 증거 — 1 인 창업자가 LLM 을 실제로 어떻게 운용했는지 박제.

### 영향
- **task 시작 비용 증가** — 다단계 mini-task 진입 전 30 분 ~ 1 시간 명령서 작성 의무.
- **`docs/redesign/MINI-*.md` 폴더 누적** — 향후 패턴 카탈로그/회고 자료로 정리 필요 (단기 적용 사례 축적 후 중기 회고).
- **갱신 책임** — task 진행 중 결정 바뀌면 명령서도 같이 수정해야 함. 안 하면 stale 명령서가 잘못된 정보 전달 위험.
- **commit message 가이드 의존성** — 단계 단위 commit 시 message body 에 단계 요약 박는 운영 습관 필요. 빠지면 옵션 D 의 한쪽 다리가 약해짐.

## Future Work

### 단기
- **다음 mini-task 박제 시 ADR-0021 적용** — 적용 트리거 (3+ 단계 / 30 분+) 충족 시 즉시 명령서 박제 시작.
- **명령서 템플릿 표준화** — 5 요소 (제목 / 전제 / 단계 / 검증 / 게이트 / 보안) 을 골격으로 한 템플릿 파일을 `docs/redesign/_TEMPLATE-MINI.md` 같은 위치에 박기.
- **`docs/redesign/MINI-*.md` 패턴을 명령서 표준 위치로 명문화** — 향후 어디에 박을지 헤매지 않도록 README/CONTRIBUTING 에 위치 명시.

### 중기
- **적용 사례 누적 시 회고** — 5 ~ 10 건 누적 후 효용 (단절 복구 성공 사례 vs 명령서 작성 비용) 정량 회고.
- **Claude Code 컨텍스트 보존 메커니즘 개선 추적** — 영구 메모리 / 자동 컨텍스트 압축 개선 도입 시 본 정책 재검토.
- **AI agent 감독 패턴 일반화** — 다른 1 인 창업자/소규모 팀에 공유 가능한 형태로 박제 (블로그 글, 포트폴리오 자료 등). ADR-0021 자체가 case study 가 됨.

## Notes

### Claude Code 가 관찰한 것 (이번 세션 한정)
- LEE 의 명령서가 다음 요소를 일관되게 포함:
  1. **전제 조건 명시** ("환경변수 X 가 길이 Y 이고 prefix Z 여야 함" 등 검증 명령 동반).
  2. **단계 번호 + 명시적 진입 트리거** ("단계 1부터", "재실행" 등 한 마디 트리거).
  3. **단계별 보고 시점** ("단계 1-1 통과 후 보고" / "각 spec 단위 통과 시 보고").
  4. **보안 원칙** ("비밀번호/key 출력 금지", "secret 발견 시 즉시 중단").
  5. **자동 진행 vs 승인 게이트** ("자동 진행 OK 는 X 까지, 그 후엔 LEE 보고").
- 작업 단위 (mini-task / 페이즈 / 단계) 마다 위 5 요소를 박은 명령서가 chat 으로 던져짐.
- Claude Code 는 단계 종료마다 결과 + 다음 단계 명령 대기 보고로 응답.

### Claude Code 가 모르는 것
- 단절 직전 LEE 가 어떤 결정을 했는지 (세션 종료 / 컨텍스트 한계 도달 / 의도적 분리).
- 단절 직후 LEE 가 복구 명령서를 작성하는 비용 / 시간 / 부담.
- 본 패턴이 다른 mini-task 에서도 일관되게 적용됐는지 (Claude Code 는 한 세션 내 사건만 관측).

### 메타 회귀 (Self-referential application)
- **이 ADR 자체가 적용 사례** — 본 ADR 작성 흐름 자체가 long-running task 의 한 형태:
  1. **Claude Code 골격 작성** (Status / Context / 빈 placeholder 섹션) → file 박제.
  2. **LEE 보충 자료** chat 메시지로 던짐 → file 갱신.
  3. **단계별 검증** (각 ADR 별 Accepted 확인 후 다음 ADR 진입).
- 즉 **"ADR 작성 task" 도 long-running task 의 한 종류** — 6 개 ADR 작성 = 6 단계 mini-task.
- **메타 회귀**: ADR-0021 작성하면서 ADR-0021 패턴 (파일 박제 + 단계별 보고 + 승인 게이트) 을 그대로 따르고 있음. 본 ADR 의 정책이 자기 자신의 작성 과정에 이미 적용됨 — 정책의 일관성/실행 가능성에 대한 가장 강한 증거.
