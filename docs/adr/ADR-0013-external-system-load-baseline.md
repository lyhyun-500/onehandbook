# ADR-0013: 외부 시스템 부하 베이스라인 정의

- **Status**: Accepted
- **Date**: 2026-05-06
- **Related**: [TS-001](../troubleshooting/TS-001-self-ddos-runbundled-incident.md)

---

## Context

### 발견 경위
- TS-001 사고 처방 배포 후, 2026-05-04 ~ 05-06 production 모니터링 진행
- Supabase usage 차트에서 **매일 04:00 UTC (= 13:00 KST)** 큰 스파이크 관찰
- 본인 직감: "AWS daily report 작업 때문 아닐까"
- 데이터 검증: 매일 04:00 UTC 정확히 1건 INSERT + 큰 read 발생
- 확인 결과: AWS Lambda/EC2의 daily report 작업이 Supabase에 read 다수 + 1 INSERT 수행

### 문제 정의
- 베이스라인이 정의되지 않으면 **정상 패턴**과 **이상 패턴**을 구분할 수 없다
- TS-001 같은 사고 시 진단을 늦추는 노이즈가 됨
- 모니터링 알람을 켜면 매일 13:00 KST에 false positive 발생 예상

---

## Decision

### 베이스라인 정의
| 시간대 (KST) | 패턴 | 출처 |
|---|---|---|
| 13:00 ~ 14:00 | 큰 read 스파이크 + 1 INSERT | AWS daily report (외부 batch) |
| 그 외 시간 | 사용자 활동 비례 | consumer traffic |

- **13:00 KST 시점 큰 스파이크는 정상 패턴으로 인식**
- 다른 시간대의 큰 스파이크만 진단 대상으로 분류
- 13:00~14:00 KST의 read 추정치: 약 5K~10K request (현재 관찰값 기준, 추후 정밀화 가능)

### 운영 적용
- 모니터링 알람 임계값 설정 시 13:00~14:00 KST 윈도우 제외 또는 baseline 별도 적용
- 향후 외부 시스템 추가 시 이 ADR에 시간대별 패턴 추가 (외부 batch 카탈로그 역할)

---

## Alternatives Considered

### 옵션 A — 베이스라인 정의 안 함 (현 상태)
- 별도 작업 없이 모니터링만 운영자 직감으로 해석
- 거부 근거: 운영자 1인 의존. 본인이 잠시 떠나거나 인계 시 13:00 스파이크 = 사고 오인 위험. TS-001 진단 중에도 노이즈로 작용했음.

### 옵션 B — 베이스라인 정의 + 카탈로그화 ⭐ 채택
- 시간대별 정상 패턴을 ADR로 박제
- 외부 batch 추가 시 카탈로그 확장
- 비용: ADR 갱신 부담 소량. 외부 batch가 자주 바뀌지 않아 실제 부담 미미.

### 옵션 C — AWS daily report 작업을 다른 시간대로 이동
- 13:00 KST를 사용자 트래픽 최저 시간대(예: 04:00 KST)로 옮겨서 충돌 회피
- 거부 근거: AWS report가 사용자 분석 성능에 실측 영향 없음 (read 위주). 회피 자체가 목적이 아니라 인지가 목적. 옵션 B가 본질에 더 부합.

### 옵션 D — 모니터링 알람을 시간대 인식형으로 구현
- 알람 시스템에 baseline schedule을 코드로 주입
- 거부 근거: 향후 단계로 적합 (Future Work에 기술). 1단계는 ADR로 인지 공유, 2단계가 코드화. 한 번에 가면 over-engineering.

---

## Revisit Criteria

다음 신호가 누적되면 ADR-0013 재검토:

1. **외부 batch 추가/일정 변경**
   - 새 시스템이 Supabase에 정기 부하를 가하기 시작할 때 → 카탈로그 확장
2. **13:00 KST 부하가 사용자 분석 성능에 영향**
   - 분석 응답 시간 측정값이 13:00~14:00 윈도우에서 baseline보다 유의미하게 늦어지면 옵션 C(시간대 이동) 재검토
3. **운영 dashboard 정밀화 필요**
   - 관찰값 기반 추정치(5K~10K)가 운영 판단에 부족할 때 → 정밀 측정 + ADR 수치 갱신
4. **외부 batch 카탈로그 5건 이상 누적**
   - ADR 1개로 관리 부담 ↑ → Future Work의 IaC 알람 룰 코드화로 승격

재검토 시 v2 업데이트, 폐기 시 superseded 표시.

---

## Consequences

### 긍정
- 정상 vs 비정상 구분이 명확해져 운영 가시성 향상
- TS-001 같은 사고 진단 시 13:00 시점 노이즈 즉시 배제 가능
- 모니터링 알람 false positive 제거

### 영향
- 외부 batch 일정 변경 시 ADR 갱신 의무 (운영 부담 소량)
- 베이스라인 추정치는 관찰값 기반 → 정밀도 한계 있음 (필요 시 정밀화)

---

## Future Work

### 단기
- 운영 dashboard에 시간대별 베이스라인 라인을 그래프로 표시
- 시간대별 비정상 임계값 정의 (예: 13:00 윈도우는 평균±2σ, 그 외는 평균±3σ)

### 중기
- 외부 시스템 부하 패턴 카탈로그 확장 (현재 1건 → 향후 추가 batch들)
- 알람 룰 코드화 (시간대별 임계값을 IaC로 관리)

---

## Notes

이 ADR의 본질은 **"정상은 무엇인가"를 명문화**하는 것. TS-001 진단 중 본인이 느낀 "13:00 스파이크는 AWS report라 무관할 것 같다"는 직감을 데이터로 검증해 박제했다.

향후 운영자가 바뀌거나 본인이 잠시 떠나도, 13:00 스파이크 보고 놀라는 일이 없도록 하는 것이 이 ADR의 존재 이유.
