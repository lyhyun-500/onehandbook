## 2. Phase 별 결정 요약

### Phase 1: 색상 토큰 인프라

**의도**:
- Tailwind v4 의 `@theme inline` 블록에 `--color-sidepanel-*` CSS 변수 정의
- 향후 Phase 2 작업에서 일관된 톤앤매너 적용

**실제**:
- `--color-sidepanel-bg`, `border`, `fg`, `card`, `accent` 등 12개 토큰 정의
- 기존 `@theme inline` 블록에 통합 (별도 블록 안 만듦) — 검증 단계에서 발견한 교훈

**핵심 학습**:
- Tailwind v4 에서 `@theme inline` 블록은 1개로 유지해야 빌드 안정
- 새 블록 만들면 유틸리티 생성이 꼬일 수 있음

### Phase 2-A: 인물 데이터 모델 확장

**의도**:
- `Character` 타입에 `summary?: string` 필드 추가
- `SIDEPANEL_CHARACTER_ROLES` 상수 정의 (5종)
- `normalizeRoleForSidePanel()` 매핑 함수 추가 (legacy "조력자" → "조연")

**실제**:
- 데이터 모델 + 상수 정의 완료
- **UI 적용은 보류** (Phase 2-B/C 에서 점진 적용)
- 결과적으로 `SIDEPANEL_CHARACTER_ROLES` 가 **2-B 까지 "예약된 코드" 상태**

**핵심 학습**:
- "준비 단계" 와 "적용 단계" 분리는 안전한 점진 마이그레이션 방식
- 단, 적용 단계 전까지는 **죽은 코드처럼 보임** → ADR/주석으로 의도 명시 필요

### Phase 2-B: 역할 옵션 5종 통일

**의도**:
- 작품설정 (`CHARACTER_ROLES` 3종) ↔ 사이드 패널 (`SIDEPANEL_CHARACTER_ROLES` 5종) 통일
- Phase 2-A 의 5종 구조 활용

**실제**:
- 옵션 A (사이드 패널 기준 5종) 채택
- `CHARACTER_ROLES` 를 deprecated alias 로 유지 (호환성)
- legacy "조력자" DB 데이터는 보존, UI 표시만 "조연" 으로 매핑

**핵심 학습**:
- DB 데이터 변환 vs 표시 시점 매핑 — **표시 시점 매핑이 더 안전** (legacy 데이터 손실 0)
- "deprecated alias 유지" 패턴으로 호환성 + 진화 동시 달성

### Phase 2-C: 카드 분리 + summary 노출

**의도**:
- `CharactersTab.tsx` (416줄 단일 파일) 분리
- summary 필드 UI 노출 (Phase 2-A 의도 실현)
- Phase 1 색상 토큰 적용

**실제**:
- 옵션 C (풀 재설계) 채택
- `CharacterCard.tsx` (194줄) + `CharacterCardFields.tsx` 신규 분리
- summary 접힘 상태에서도 노출 (line-clamp-2)
- 색상 토큰 적용 (`--color-sidepanel-danger` 1개만 신규 추가)
- 정적 디자인까지만 (애니메이션은 2-D 영역)

**발견된 버그 2개**:
1. `border` shorthand vs `borderLeftColor` 충돌 → `inset box-shadow` 로 해결
2. 새 인물 생성 시 select ↔ 헤더 뱃지 불일치 → role 기본값 명시로 해결

**핵심 학습**:
- 풀 재설계 (옵션 C) 가 옵션 A/B 보다 **이중 작업 회피** 측면에서 EV 우세
- 정적 디자인 / 동적 효과 영역 분리 명시가 작업 범위 보호에 결정적
- 200줄 이내 컴포넌트 제한이 자연스러운 책임 분리 강제

## 3. 패턴화된 학습

### 학습 1: "준비 단계" 와 "적용 단계" 분리의 가치

```
Phase 2-A: 데이터 모델 확장 (사용자 영향 0)
  ↓
Phase 2-B: 역할 통일 (간접 영향)
  ↓
Phase 2-C: UI 재설계 (직접 영향)
```

**왜 좋은가**:
- 각 단계 빌드/테스트 부담 작음
- 회귀 위험 분산
- 중간에 멈춰도 직전 상태 안정

**전제 조건**:
- 의도 문서화 (ADR 또는 주석)
- 적용 단계까지의 시간 추정
- "예약된 코드" 가 6개월 이상 방치되면 위험 (잊힘)

### 학습 2: legacy 데이터 보존 + 표시 매핑

```
DB 데이터 직접 변환 ❌ (legacy 손실, 롤백 어려움)
  ↓
표시 시점 normalizeXxx() 매핑 ✅ (안전, 점진 마이그레이션)
```

**원칙**:
- DB 는 신중히, UI 는 유연히
- 매핑 함수 위치 = "데이터 → UI" 경로의 가장 가까운 곳
- 매핑 결과를 **저장값** 으로 사용하지 않을 것 (저장 시 raw 사용)

### 학습 3: 풀 재설계가 점진 변경보다 나을 때

EV 분석 (Phase 2-C):
```
옵션 A (최소): summary 만 추가     → P(win) 0.95, EV 약한 양수
옵션 B (중간): 컴포넌트 분리 + summary → P(win) 0.85, EV 중간 양수
옵션 C (풀):  재설계 + summary + 토큰 → P(win) 0.85, EV 강한 양수
```

**옵션 C 가 우세한 조건**:
- 같은 영역에서 Phase 2-D 가 또 손댈 예정
- 이중 작업 비용 > 한 번에 큰 변경 비용
- 빌드/테스트 인프라가 안정적

### 학습 4: "정적 vs 동적" 영역 명시의 보호 가치

Phase 2-C 지시서에 명시:
```
정적 디자인 (이번 범위):
- 컴포넌트 분리, 색상 토큰, 정보 배치, 상태 시각 신호 (정적)

동적 효과 (Phase 2-D 영역):
- 애니메이션, 호버 디테일, 마이크로 인터랙션
```

**왜 중요했나**:
- Cursor 가 "더 좋게 만들고 싶음" 으로 영역 침범 가능성
- 명시 없으면 작업 범위 자연 확장 (스코프 크리프)
- 영역 분리 = Phase 2-D 의 작업 보호

## 4. EV 프레임의 검증

ADR 시리즈 전체에서 EV 공식 적용:

| ADR | 영역 | 옵션 수 | 감 vs EV |
|-----|------|--------|---------|
| 0001 NAT 모델 | 비즈니스 | 3 | 일치 |
| 0002 통합 저장 v2 | 기술 | 2 | 일치 |
| 0003 어드민 위치 | 기술 | 3 | 불일치 (EV 우선) |
| 0004 패들 시점 | 비즈니스+법무 | 4 | 일치 |
| 0005 useState props | 기술 버그 | 3 | 일치 |
| 0006 알림 read_at | 기술 버그 | 3 | 일치 |
| **(이번 회고)** | **메타** | **N/A** | **N/A** |

**검증된 것**:
- EV 프레임이 **아키텍처 / 비즈니스 / 버그 수정** 모두 적용 가능
- 6연속 적용으로 **방법론 자체가 자산화**
- "감 ≠ EV" 인 케이스 (ADR-0003) 도 발견 → EV 가 검증 도구로 작동

**EV 공식 진화**:
```
v1 (ADR-0001~0003): E = P(win) × Gain − P(loss) × Loss
v2 (ADR-0004~): + 비즈니스/법무 비기술 영역 확장
v3 (ADR-0005~): + 미시적 구현 결정 (key remount, RPC 호출 등)
```

## 5. Phase 2-D 권장 사항

### 작업 범위 (이번 ADR 에서 미리 정의)

**핵심 영역**:
- 펼치기/접기 애니메이션 (slide / fade)
- hover 효과 (subtle border / background 변화)
- focus ring 정교화 (현재 임시 amber 사용 중)
- 마이크로 인터랙션 (저장 버튼 ripple 등)

**비핵심 (보류)**:
- 그라디언트 (Novel Agent 톤앤매너 일관성 위해)
- 페이지 전환 애니메이션 (사이드 패널 안 영역)

### 우선 처리 항목

1. **임시 amber 포커스 링 → 토큰화**
   - Phase 2-C 에서 임시 사용 중
   - `--color-sidepanel-focus` 토큰 정의 + 적용

2. **펼치기 애니메이션**
   - 현재 즉시 전환 → 200~300ms ease-out
   - height auto 트랜지션의 React 패턴 주의

3. **dirty dot / invalid stripe 폴리싱**
   - 현재 정적 표시
   - 펄스 또는 페이드인 (subtle)

### 작업 규모 추정

- 1~3시간 (대부분 CSS 작업)
- 신규 파일 0~1개
- 빌드 검증 1~2회

### 권장 옵션 명시 (지시서 참고용)

```
옵션 A: 핵심만 (애니메이션 + focus 토큰화) — 1시간
옵션 B: 핵심 + dirty/invalid 폴리싱 — 2시간
옵션 C: 풀 폴리싱 (마이크로 인터랙션 포함) — 3시간

추천: 옵션 B (핵심 + 시각 신호 강화)
```

## 6. 다른 영역 적용 가능 학습

### 어드민 영역에 적용

**현재 어드민 상태** (4/25 기준):
- 유저 관리 ✅ Live
- 탈퇴 로그 ⏳ 진행 중
- 문의 관리 ⏳ 대기

**Phase 2 학습 적용 가능**:
- 학습 1 (준비 → 적용 분리): 새 어드민 페이지마다 데이터 모델 먼저, UI 다음
- 학습 4 (영역 명시): "기능 구현" vs "디자인 폴리싱" 분리
- 풀 재설계 vs 점진: 어드민은 단순한 CRUD 라 점진이 우세

### 분석 파이프라인 영역에 적용

**잠재 작업**:
- 분석 리포트 UI 개선
- 일괄 분석 진행 표시 강화
- 작가 피드백 수집 강화

**Phase 2 학습 적용**:
- 학습 2 (legacy 보존): 기존 분석 결과 데이터 형식 유지하면서 UI 만 개선
- 학습 3 (풀 재설계): 분석 리포트는 자주 손대지 않으니 풀 재설계 우세 가능


- Phase 2-B, 2-C 에서 5번 검증

- AI 가 자율 진행 시도해도 게이트에서 멈춤 → 안전

- 5파일 이상 변경 예상 시 계획 먼저 보고
- 작업 폭 폭증 방지
- 옵션 C 풀 재설계 시 자연스럽게 트리거됨

### 3. "정적 vs 동적" 영역 명시
- Phase 2-C 에서 처음 도입
- 다음 ADR (0007 회고) 에서 패턴화
- 향후 작업 지시서 표준 항목

### 4. 의도적 "예약된 코드"
- Phase 2-A 의 SIDEPANEL_CHARACTER_ROLES
- Phase 2-C 에서 활용
- ADR/주석 명시 필수

## 8. 개선 제안 (다음 스프린트)

### 잘된 것 (유지)
- ADR 시리즈 일관성 (제목 / Status / Revision History 형식)
- 단계 분리 (Phase 1 → 2-A → 2-B → 2-C)
- 의도 문서화 (각 Phase 의 "왜")

### 개선이 필요한 것

**1. 시각 산출물 부족**
- 현재 ADR 은 텍스트 위주
- "Before / After" 스크린샷 없음
- 다음 ADR 부터 가능하면 캡처 첨부

**2. 시간 추정 정확도**
- Phase 2-C: 예상 1.5시간 → 실제 2.5시간 (버그 2개 발견 포함)
- 1.5배 버퍼를 기본으로 잡아야 정확

**3. 빌드 검증 자동화 부재**
- 매번 LEE 가 수동 빌드 결과 확인
- 향후: pre-push hook 또는 GitHub Actions

**4. 회귀 테스트 자동화 부재**
- Phase 2-C 후 발견된 버그 2개 = 수동 테스트로 발견
- 향후: e2e 테스트 (Playwright 등) 검토

**5. 디자인 시스템 부재**
- Phase 1 토큰 정의했지만 **사용 가이드 없음**
- 어떤 토큰을 어디에 쓰는지 문서화 필요
- 다음 ADR 후보: "Novel Agent 디자인 시스템 확립"
