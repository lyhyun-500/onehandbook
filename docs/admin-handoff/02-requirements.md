# 02. Requirements — 기능 명세 (MVP)

---

## 🎯 MVP 범위

아래 4개 기능 페이지를 이번 스프린트에서 구현.

| 페이지 | 우선순위 | 경로 |
|-------|--------|-----|
| **어드민 홈 (대시보드)** | ⭐⭐⭐ | `/admin` |
| **유저 관리** | ⭐⭐⭐ | `/admin/users` |
| **NAT 관리 (유저 상세 내부)** | ⭐⭐⭐ | `/admin/users/[id]` |
| **문의 관리** | ⭐⭐⭐ | `/admin/inquiries` |
| **탈퇴 로그** | ⭐⭐ | `/admin/withdrawals` |

---

## 1. 어드민 홈 (`/admin`)

### 목적
- 로그인 후 첫 진입점
- 주요 지표 한눈에
- 주요 페이지로 네비게이션

### 화면 요소

#### 상단 지표 카드 (4개, 가로 배치)
1. **전체 유저 수** — 탈퇴 제외
2. **오늘 가입자** — 오늘 00:00 이후
3. **미답장 문의 수** — 답장 상태 false 건
4. **이번 주 탈퇴 수** — 최근 7일

#### 빠른 이동 카드 (3개)
1. 유저 관리 → `/admin/users`
2. 문의 관리 → `/admin/inquiries`
3. 탈퇴 로그 → `/admin/withdrawals`

#### 최근 활동 (하단)
- 최근 NAT 조정 로그 5건 (어떤 유저에게, 얼마만큼, 누가)
- 최근 미답장 문의 5건 (이메일, 제목, 시간)

### 데이터 소스
- `users` (count, 가입일 필터)
- `inquiries` (답장 상태 필터)
- `withdrawal_logs` (7일 필터)
- `nat_adjustments` (신규 테이블, §3 에서 정의)

### 에러 처리
- 데이터 로딩 실패 시 "데이터를 불러올 수 없습니다. 새로고침해주세요" 표시
- 각 카드 단독 실패 가능 (전체 페이지 깨지지 않게)

## 2. 유저 관리 (`/admin/users`)

### 목적
- 유저 검색 + 목록 조회
- 유저별 주요 정보 요약
- 유저 상세 페이지 진입

### 화면 요소

#### 상단 검색 바
- **검색 필드**:
  - 이메일 (부분 일치)
  - 닉네임 (부분 일치)
- **필터**:
  - 가입 경로 (Google / Naver / 전체)
  - 활성 상태 (활성 / 탈퇴 / 전체)
- **정렬**:
  - 가입일 (최신/오래된 순)
  - NAT 잔량 (많은/적은 순)
- **검색 버튼** + **초기화 버튼**

#### 결과 테이블 (columns)
| 컬럼 | 내용 |
|-----|-----|
| 닉네임 | 유저 닉네임 |
| 이메일 | 로그인 이메일 |
| 가입일 | YYYY-MM-DD |
| 로그인 방식 | Google / Naver 뱃지 |
| NAT 잔량 | 숫자 |
| 작품 수 | 소프트 딜리트 제외 |
| 분석 횟수 | analysis_jobs count |
| 상태 | 활성 / 탈퇴 뱃지 |
| 액션 | "상세" 버튼 |

#### 페이지네이션
- 기본 50건씩
- "더보기" 버튼 방식 또는 "< 1 2 3 >" 방식 (둘 중 Claude Code 판단)

### 동작
- 검색 결과 없을 때: "검색 결과가 없습니다" 메시지
- 탈퇴 유저도 검색 가능 (상태 필터 "탈퇴" 또는 "전체")
- 테이블 행 클릭 또는 "상세" 버튼 → `/admin/users/[id]` 이동

### 데이터 소스
- `users` + `works` (count) + `analysis_jobs` (count)
- N+1 쿼리 피하기 위해 JOIN 또는 집계 쿼리 사용

## 3. 유저 상세 (`/admin/users/[id]`)

### 목적
- 특정 유저의 모든 맥락 조회
- NAT 수동 조정 (충전/차감)
- 조정 히스토리 확인

### 화면 구성

#### 상단 유저 정보 카드
- 닉네임, 이메일, 가입일, 로그인 방식, 활성 상태
- 약관 동의 여부 (이용약관, 개인정보, 마케팅)
- 탈퇴 유저라면: 탈퇴일 + 탈퇴 이유 표시

#### 중단: NAT 관리 섹션 ⭐ 핵심
##### 현재 잔량 표시
- 큰 숫자로 "현재 NAT 잔량: 45"

##### NAT 조정 폼
- **조정 유형**: 충전 / 차감 (라디오 또는 셀렉트)
- **수량**: 숫자 입력 (양수만)
- **사유**: 필수 입력 텍스트 (50자 이하)
  - 예: "이벤트 보상", "환불 요청", "시스템 오류 복구"
- **조정 버튼**: 클릭 시 **확인 모달**
  - "정말 이 유저에게 50 NAT 를 충전하시겠습니까?"
  - "확인" / "취소"
- 성공 시 토스트: "50 NAT 충전 완료"

##### NAT 조정 히스토리
- 최근 20건 표시
- 컬럼: 시간, 유형(충전/차감), 수량, 사유, 조정자, 조정 후 잔량

#### 하단: 활동 내역 섹션
##### 작품 목록
- 테이블: 제목, 장르, 회차 수, 최근 업데이트, 상태(활성/삭제)
- 삭제된 작품도 표시 (회색)

##### 최근 분석 기록
- 테이블: 작품, 회차/유형, 분석일, NAT 소모, 상태(성공/실패)
- 최근 10건

##### 문의 기록
- 이 유저가 보낸 1:1 문의 목록
- 제목, 내용 요약, 상태(답장 여부), 보낸 시간

### 에러 처리
- 존재하지 않는 user id: 404 페이지
- NAT 조정 실패 시: 원래 잔량으로 복구 + 에러 메시지
- 조정 중 다른 관리자가 동시 수정: (베타는 무시, LEE 1명이라 충돌 없음)

## 4. 문의 관리 (`/admin/inquiries`)

### 목적
- 1:1 문의 목록 조회
- 답장 완료 상태 표시/변경
- 답장 누락 방지

### 화면 요소

#### 상단 필터
- **상태**: 전체 / 미답장 / 답장 완료
- **기간**: 전체 / 최근 7일 / 최근 30일
- **검색**: 제목 키워드

#### 목록 테이블
| 컬럼 | 내용 |
|-----|-----|
| 상태 뱃지 | 미답장(빨강) / 답장완료(회색) |
| 보낸 유저 | 닉네임 + 이메일 |
| 제목 | 문의 제목 |
| 내용 요약 | 첫 100자 |
| 보낸 시간 | 상대 시간 (예: 2시간 전) |
| 액션 | "답장함 / 답장취소" 토글 버튼 |

#### 문의 상세 드로어 (행 클릭 시)
- 전체 내용 표시
- 유저 정보 (닉네임, 이메일, 가입일)
- "이 유저 상세 보기" 링크 (→ `/admin/users/[id]`)
- 답장 상태 변경 버튼
- **실제 답장 발송은 이메일 앱에서** (Gmail 등)
  - 복사 버튼: 유저 이메일 복사
  - 복사 버튼: 문의 내용 복사

### 동작
- 답장 상태 변경 시 확인 모달 생략 (되돌리기 쉬우니까)
- 상태 변경 후 토스트: "답장 완료로 변경됨"

### 데이터 소스
- `inquiries` (신규 테이블 또는 기존 활용)
- Resend 발송 이력은 MVP 에서 연동 안 함 (LEE 수동 표시)

## 5. 탈퇴 로그 (`/admin/withdrawals`)

### 목적
- 탈퇴 유저의 이유 패턴 관찰
- UX 개선 인사이트 수집

### 화면 요소

#### 상단 요약
- 최근 7일 탈퇴 수
- 최근 30일 탈퇴 수
- 전체 탈퇴 수

#### 목록 테이블
| 컬럼 | 내용 |
|-----|-----|
| 탈퇴일 | YYYY-MM-DD HH:MM |
| 닉네임 | 탈퇴 시점 닉네임 |
| 이메일 | 이메일 (일부 마스킹 선택) |
| 가입 경로 | Google / Naver |
| 이용 기간 | 가입일 → 탈퇴일 (일수) |
| 탈퇴 이유 | 텍스트 (있으면 전문, 없으면 "미입력") |
| 작품 수 | 탈퇴 시점 작품 수 |
| 분석 횟수 | 탈퇴 시점 분석 횟수 |

#### 필터
- 기간: 전체 / 7일 / 30일 / 90일

### 동작
- 단순 조회만 (편집 없음)
- 정렬: 탈퇴일 역순 (최신 먼저)
- 페이지네이션: 50건씩

### 데이터 소스
- `withdrawal_logs`

## 🚫 범위 제외 (중요)

### 이번 MVP 에 포함 안 함

| 기능 | 이유 |
|-----|------|
| 통계 대시보드 (DAU/MAU 차트) | 다음 스프린트 |
| Paddle 결제 UI | 사업자 등록 후 별도 작업 |
| 공지사항 CMS | 필요성 낮음 |
| 분석 job 재시도 | 다음 스프린트 |
| 작품/회차 편집 | 리스크 높음 (유저 데이터 직접 수정) |
| 이메일 대량 발송 | 별도 도구 (Mailchimp 등) 사용 |
| 권한 세분화 (role 다층화) | 지금은 admin/user 2단계만 |
| 2FA / IP 제한 | 베타 단계 과도함 |

### 질문 있으면?
- Claude Code 가 "이것도 추가할까요?" 라고 제안 → **대부분 NO**
- 범위 확장 = LEE 의 시간 잠식
- "나중에" 라고 답하면 다음 스프린트 백로그로 기록만

## ✅ 완료 기준 (Definition of Done)

각 페이지가 아래 조건 모두 만족 시 완료:

1. 목적 달성 (이 문서의 "목적" 항목)
2. 에러 처리 구현 (데이터 없음, 로딩 실패 등)
3. 반응형 불필요 (데스크탑 전용, 최소 너비 1200px 가정)
4. 권한 차단 확인 (로그아웃 상태 접근 시 차단)
5. 빌드 통과 (`npm run build`)
6. 스크린샷 1장 첨부 (작업 완료 보고 시)
