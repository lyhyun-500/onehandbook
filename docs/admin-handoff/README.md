## 🎯 목표 한 줄 요약

**LEE 가 현재 SQL 로 수동 처리하는 운영 업무를 UI 로 전환.**

구체적으로:
- 유저 정보 조회 + NAT 수동 충전/차감 + 그 로그
- 1:1 문의 목록 + 답장 상태 관리
- 탈퇴 로그 조회

### 소비자 사이트 코드 건드리지 마
- `onehandbook/src/app/(main)/**`
- `onehandbook/src/app/studio/**`
- `onehandbook/src/app/api/analyze/**`

### 기존 DB 테이블 변경 금지
- 신규 테이블 추가는 OK
- 기존 테이블에 컬럼 추가는 **LEE 승인 후에만**

### Git push 금지
- 커밋까지만 자동, push 는 LEE 수동

### 환경변수 파일 직접 수정 금지
- `.env`, `.env.local` 등 건드리지 말 것
- 새 환경변수 필요하면 "이 이름으로 추가 필요" 보고만

## ✅ 작업 흐름 (권장)
Step 1. 문서 전체 숙독
Step 2. LEE 에게 "이해한 내용" 요약 보고
Step 3. 첫 작업 범위 제안 (예: "어드민 레이아웃 + 권한 미들웨어 먼저")
Step 4. LEE 승인 대기
Step 5. 구현 (작은 단위로)
Step 6. 작업 단위마다 빌드 검증 + 보고
Step 7. LEE 확인 후 다음 단계

**❌ 안 되는 흐름**: 문서 읽자마자 전부 구현 시작
**✅ 되는 흐름**: 이해 → 계획 → 승인 → 작은 단위 → 보고 → 반복

## 🧭 설계 우선순위

LEE 의 가치관 기준 우선순위:

1. **작동하는 것** (기능 완성도)
2. **안전성** (운영 데이터 다루므로 실수 방지 UI)
3. **포트폴리오 가치** (스크린샷 남길 수 있게)
4. **확장성** (나중에 CS 담당자 영입 대비)
5. **디자인** (기능 우선, Supabase 스타일로 충분)

## 📞 중간 질문 생기면

LEE 는 웹 Claude (대화용) 와 또 다른 채팅을 통해 병렬로 작업 중.
Claude Code 가 중간에 판단 어려운 지점 만나면:

1. 작업 멈추고 보고
2. 질문 구체화 (2~3개 선택지로)
3. LEE 답변 대기
4. 재개

## 🎓 이 프로젝트의 특수성

- **1인 운영** — 코드 리뷰어 없음, Claude Code 자신이 품질 관리

- **바이브 코딩** — LEE 는 AI 지시 + 검증 방식 선호
